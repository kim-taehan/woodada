# woodada (우다다) — 개발 프롬프트 / 스펙

사다리 게임처럼 누구나 바로 돌릴 수 있는 **동물 경주 추첨 게임**을 만들어줘.
귀여운 동물들이 원형 트랙을 달려 순위를 정하고, 그 순위를 상품·벌칙 추첨에 쓸 수 있다.

1차 목표는 브라우저에서 바로 열리는 웹 게임이다. 단, 나중에 **모바일 앱**과 **멀티플레이어**로 확장하고, **캐릭터를 계속 추가**할 것이므로 아래 [설계 원칙]을 반드시 지켜 구조를 짠다.

---

## 0. UI/UX 원칙 (최우선 — 다른 모든 기능보다 우선한다)
- 사다리 게임의 본질은 "이름 넣고 → 돌리고 → 결과 본다"이다. 앱을 열면 **3초 안에 경주를 시작**할 수 있어야 한다.
- 첫 화면에는 **[참가자 이름 입력]과 [출발] 버튼만** 보인다. 그 외 모든 설정은 기본값으로 처리되거나 접혀 있다.
- 합리적 기본값 — 아무 설정도 건드리지 않아도 바로 플레이된다:
  - 캐릭터: 기본 3종 중 랜덤 자동 배정 · 모드: 개인전 · 결과 매핑: 없음(순위만) · 스킨: 기본 외형
- 진보적 공개(progressive disclosure): 캐릭터 선택, 팀 구성, 결과 매핑, 스킨, 멀티플레이어는 모두 '원하면 펼치는' 선택 영역에 둔다. 기본 경로를 절대 방해하지 않는다.
- 빠른 입력: 이름은 엔터로 연속 추가하거나, 여러 줄을 한 번에 붙여넣어 일괄 등록할 수 있다.
- 빠른 반복: 결과 화면에서 '다시하기'로 같은 멤버 재경주가 한 번에 된다. 카운트다운은 스킵 가능.
- 경주 길이는 지루하지 않게 짧게(수 초~십수 초)로 기본 설정한다.
- 모바일 우선: 한 손, 세로 화면, 탭 위주로 모든 핵심 동작이 가능해야 한다.

---

## 1. 핵심 컨셉
- 귀여운 동물 캐릭터들이 원형 트랙을 달려 순위를 겨룬다.
- 각 캐릭터는 고유 스킬을 가지며 경주 중 **자동으로(랜덤 타이밍)** 발동된다. 플레이어 조작은 없고 매 판 결과가 달라진다.
- 스킬 덕분에 끝까지 역전이 일어나 누가 1등인지 모르게 한다.
- 톤: **귀여움 × 코믹·병맛.** 둥글둥글 귀여운 외형이 과장된 표정·슬랩스틱과 만나 "귀여운데 한심한" 갭으로 매력을 만든다.

## 2. 캐릭터 시스템 (확장 가능 — 기본 3종으로 시작해 점진적으로 부가)

### 2.1 기본 캐릭터 3종 (1차 활성)
1차 개발에서는 **강아지·토끼·원숭이 3종**을 활성화한다. 외형 레퍼런스 SVG가 각각 있다(`parts` 애니 구현 소스).

- **🐶 강아지 — 우다다(Zoomies) · [부스트·변칙] · (게임 마스코트)**
  - **스킬**: 이유 없이 갑자기 **통제 불능으로 폭주**한다. 순식간에 앞으로 튀어나가지만, 방향이 들쭉날쭉해 가끔 옆길로 샌다.
  - **성격**: 산만함의 화신. 신나서 혀 빼고 달린다. 게임 이름(우다다)을 회수.
  - **대사**: "우다다다다!!!"
  - **레퍼런스**: `woodada-dog.svg` / **palette**: base `#F6D9A8`, point `#E89B4C`, outline `#B5702E`, nose `#5B3A24`, cheek `#F4A6A0`, tongue `#EF8E8E`

- **🐰 토끼 — 방심 낮잠(Nap) · [변칙]**
  - **스킬**: 앞서갈수록(1등에 가까울 때) 자만해서 멈춰 꾸벅꾸벅 졸다가(머리 위 "zzz" + 정지), 화들짝 깨서 다시 폭주한다(번쩍 + 대시). 토끼와 거북이 우화 — 스스로 발목을 잡지만 깨면 폭발.
  - **성격**: 자만하는 허당. 졸다 깨다.
  - **대사**: "…zzz… (번쩍) 으아아!!!"
  - **레퍼런스**: `woodada-rabbit.svg` / **palette**: base `#F5F1EC`, point(귀안쪽·코) `#EF9BB0`, cheek `#F2A6B6`, outline `#A99CB2`

- **🐒 원숭이 — 바나나 던지기(Banana) · [방해]**
  - **스킬**: **앞 또는 뒤에 있는 캐릭터 하나를 노려 바나나 껍질을 던진다.** 맞으면 미끄러져 잠시 멈춘다(미끄덩 + 별). **단, 표적은 확률적으로 피할 수 있다**(피하면 바나나가 휙 빗나감 — 헛던짐 개그). 같은 팀 제외.
  - **성격**: 장난기 폭발 트러블메이커. 까불다 자기가 헛던지기도.
  - **대사**: "받아라! (까득)" / 빗나가면 "어… 빗나갔네"
  - **레퍼런스**: `woodada-monkey.svg` / **palette**: base(털) `#A9743C`, face/배 `#F0CDA0`, outline `#6E4421`, cheek `#E89AA0`, banana `#F5C84A`

### 2.2 추가 예정 카탈로그 (이후 데이터로 부가)
정의를 보관해 두고, 기본 3종이 안정되면 **데이터(+필요 시 스킬 핸들러)만 추가**해 하나씩 투입한다.

- **🦔 고슴도치 — 밀치기(Shove) · [방해·근접]**: 가까이 붙은 동물을 가시로 옆/뒤로 밀어낸다(같은 팀 제외). "붙지 마. 진짜."
- **🦅 독수리 — 채가기(Snatch) · [방해·변칙]**: 앞서가는 동물 하나를 낚아채 뒤에 떨군다(같은 팀 제외). "내 거다. (콱) …어, 돌이네."

> 그 외 후보(거미·문어·전갈·코끼리·다람쥐·뱀·펭귄·멧돼지·스컹크 등)도 같은 방식으로 데이터로 부가 가능하다.

### 2.3 스킬 가독성 (중요 — 안 읽히면 없는 거나 마찬가지)
경주는 빠르고 여러 캐릭터가 동시에 움직인다. 스킬이 복잡하거나 미묘하면 플레이어가 **발동했는지조차 모른다.** 따라서:
- 스킬 효과는 **단순하고 직관적**이어야 한다. 한눈에 "뭘 했는지" 읽혀야 한다.
- **안 보이는 미세 수치 조정 금지.** 효과는 과장되고 눈에 보이게(앞으로 쭉 / 멈춤 / 바나나 명중·회피 등 큰 변화).
- 발동 순간 **세 신호를 함께** 띄운다:
  1. **큰 비주얼 이펙트** (먼지·번쩍·궤적·바나나 등)
  2. **머리 위 대사 말풍선** (짧고 크게, 잠깐 떴다 사라짐)
  3. **눈에 띄는 위치·속도 변화**
- 말풍선은 귀여운 톤에 맞게 디자인한다. 동시다발로 떠도 가독성이 유지되게 한다(겹침·우선순위 처리).

### 2.4 캐릭터 데이터 스키마 + 스킬 핸들러 레지스트리 (확장의 핵심)
캐릭터를 **데이터**로, 스킬 동작을 **핸들러**로 분리한다.

- **캐릭터 데이터(JSON 예시 — 강아지)**:
```json
{
  "id": "dog",
  "name": "강아지",
  "visualType": "parts",
  "visualRef": "woodada-dog.svg",
  "proportions": { "headBody": "2.5등신", "bigEyes": true },
  "palette": {
    "base": "#F6D9A8", "point": "#E89B4C", "outline": "#B5702E",
    "nose": "#5B3A24", "cheek": "#F4A6A0", "tongue": "#EF8E8E"
  },
  "runStyle": "scattered",
  "skill": {
    "type": "zoomies",
    "cooldownMs": [3000, 6000],
    "params": { "burstMin": 40, "burstMax": 120, "strayChance": 0.25 }
  },
  "lines": { "skill": "우다다다다!!!", "win": "…", "lose": "…" }
}
```
- **원숭이 스킬 params 예시**: `{ "target": "front|back", "hitStun": 700, "dodgeChance": 0.35 }` (앞·뒤 무작위 표적, 회피 확률 포함).
- **스킬 핸들러 레지스트리**: 스킬 동작을 `type → handler`로 등록한다.
  - 핸들러 시그니처(개념): `handler(racer, allRacers, rng, params) → 효과 적용`
  - 무작위성은 주입된 PRNG(`rng`)만 사용한다.
  - 예: `SkillRegistry.register("zoomies", zoomiesHandler)`.
- **새 캐릭터 추가 규칙**: 기존 스킬 type 재사용 → 데이터 한 덩이만 추가. 새 스킬 → 핸들러 1개 + 데이터.
- 스킬 type 풀:
  - 활성(1차): `zoomies`(강아지), `nap`(토끼), `banana`(원숭이)
  - 예정: `shove`(고슴도치), `snatch`(독수리)

### 2.5 아트 디렉션 — 무조건 귀엽게 (필수)
모든 캐릭터는 귀여워야 한다. 추상적 "귀여움"을 아래 기준으로 구현한다.
- **비율**: 2~2.5등신. 큰 머리·큰 눈·작은 몸(베이비 스키마 / 치비).
- **형태**: 둥글둥글, 부드러운 곡선 위주. 날카로운 요소(가시·부리·발톱)도 **둥글게 순화**한다.
- **눈**: 크고 반짝이는 눈(하이라이트 점). 단순하되 표현력 있게 — 귀여움의 핵심.
- **컬러**: 부드럽고 따뜻한 톤. 캐릭터마다 **고유색**으로 식별(강아지 탄색 / 토끼 흰·분홍 / 원숭이 갈색·베이지).
- **표정**: 단순하되 과장되고 풍부하게. 병맛 슬랩스틱과 결합 — 귀여운 얼굴로 어이없는 짓.
- **모션**: squash & stretch, 통통 바운스, 오버슈트로 탱글탱글하게.
- **식별성 유지**: 다 귀여워도 캐릭터별 **실루엣·색·특징**(강아지 늘어진 귀, 토끼 긴 귀, 원숭이 양옆 귀+꼬리+바나나)이 한눈에 구분돼야 한다.

### 2.6 캐릭터 비주얼 / 애니메이션
- 캐릭터 외형은 `visualType`으로 데이터에서 지정한다: `parts`(파츠 절차 애니, **1차 채택**) / `sprite` / `spine` / `emoji`(폴백).
- **`parts`**: 머리·귀·눈·몸·다리·꼬리 파츠를 코드로 움직여(squash & stretch) 귀여움과 잘 맞고 에셋 부담 없이 부가가 쉽다. 렌더러가 `visualType`을 해석하므로 나중에 인기 캐릭터만 `sprite`/`spine`으로 업그레이드 가능.
- **표준 동작 세트**: `idle / run / skill(캐릭터별) / win / fall(피격·넘어짐)`.
- **외형 레퍼런스 파일**(비율·색·표정·실루엣 기준이자 `parts` 구현 소스):
  - 강아지 `woodada-dog.svg`, 토끼 `woodada-rabbit.svg`, 원숭이 `woodada-monkey.svg` (각 기본/달리기/스킬 3포즈).

### 2.7 캐릭터 공통 연출
- **달리기 폼을 캐릭터마다 다르게** 한다.
- 스킬 효과물(바나나 껍질·넘어진 상태·먼지·zzz 등)이 화면에 보이게 한다.
- **1등 세레머니와 꼴찌 망연자실 반응**도 캐릭터별로 다르게.

## 3. 밸런스
- 어떤 캐릭터도 통계적으로 유리하지 않게(승률이 고르게) 스킬 수치를 조정한다.
- 같은 캐릭터를 여러 참가자가 중복 선택해도 된다.
- 스킬 수치(쿨다운, 폭주량, 낮잠 빈도, 바나나 회피 확률 등)는 먼저 동작하게 만든 뒤 플레이해보며 튜닝한다.
- 기본 3종은 자기폭발(강아지)·자기발목(토끼)·남방해(원숭이)로 결이 갈린다. 캐릭터가 늘면 [부스트/방해/방어·변칙]이 골고루 섞이도록 카탈로그에서 선택한다.

## 4. 게임 모드 (개인전 / 팀전)
- 모드는 **설정 객체로 정의**한다(데이터 주도).
- **개인전**: 모든 참가자가 개별 순위로 겨룬다.
- **팀전**: 참가자를 팀으로 묶는다. 프리셋(2v2, 3v3, 2v2v2) + '팀 수 × 팀당 인원' 커스텀(예: 4v4, 2v2v2v2).
- **팀 승리 판정** (모드 설정에서 선택):
  - a) **순위 합산** — 팀원 등수 합이 가장 작은 팀 우승 (기본값). 합이 같으면 1등을 더 많이 낸 팀 우선.
  - b) **에이스** — 각 팀 가장 빠른 동물의 등수로 비교. 같으면 차순위로 비교.
  - 판정은 코어 결과(각 참가자 도착 순서)에서 파생되는 **'스코어링 전략'으로 분리 구현**한다(릴레이 등은 함수 추가로 확장).
- **팀 스킬 규칙**: 방해 스킬(바나나·밀치기 등)은 같은 팀에는 발동하지 않고 상대 팀에게만. 부스트·변칙은 자기 자신에게.

---

## 5. 참가자 관리
- 참가자를 2~8명까지 추가/삭제하고, 각자 이름과 캐릭터(기본 3종 중)를 고른다. 안 고르면 랜덤 배정.
- 이름을 비우면 기본 이름(예: "강아지1")을 자동 부여한다.
- 팀전에서는 참가자 데이터에 teamId를 둔다.

---

## 6. 결과 매핑 (사다리 게임의 추첨 기능)
- 토글로 '결과 입력' 모드를 켤 수 있다.
- 각 순위(1등, 2등 …)에 텍스트 결과(예: "커피 쏘기", "당첨", "꽝")를 매핑한다. 비우면 순위만 보여준다.
- 팀전에서는 팀 단위 결과 매핑도 지원한다(예: "꼴찌 팀이 쏘기").

---

## 7. 경기장 (원형 트랙 + 인코스/아웃코스 자유 주행)
- **원형(타원형) 순환 트랙.** 진짜 경마장/육상 경기장 룩:
  - 빨강·주황 **타탄 트랙 표면**, 흰색 레인 구분선(시각 참고용), 트랙 안쪽 **잔디 필드**, 바깥 **관중석 실루엣**.
  - **스타팅 게이트**(출발)와 **흰 굵은 결승선 + 결승 테이프**, **전광판/스코어보드**로 실시간 순위 표시.
- **레인은 고정이 아니다.** 동물은 **기본적으로 인코스(안쪽)**를 달리다가 **앞지르기 위해 아웃코스(바깥쪽)로 자유롭게 이동**한다(아래 [경주 진행]).
- **트랙 모양은 렌더러 책임**: 엔진은 `진행도(전후)` + `레인 위치(좌우, 0=인코스 ~ 1=아웃코스, 연속값)`만 다루고, 렌더러가 이를 타원 경로 위 좌표로 매핑한다(트랙 모양이 시뮬레이션에 영향 없음).
- **식별성**: 자유 주행이라 캐릭터가 겹칠 수 있다. **약한 원근(거의 톱다운 부감)** + **z-order 정리** + **상시 떠 있는 이름표·색 마커**로 항상 구분되게 한다.
- **랩 수**: 기본 한 바퀴(설정으로 변경 가능).

---

## 8. 경주 진행 (L1 추월 모델 — 물리엔진 없이 규칙으로)
- '출발' → 3·2·1 카운트다운 → 경주 시작.
- 각 캐릭터는 `진행도(전후)`와 `레인(좌우)`을 가진다. 매 프레임 속도가 변하고 스킬이 자동 발동된다.
- **추월/블로킹 규칙**(물리엔진 X, 게임 규칙 + PRNG):
  - 기본은 인코스 주행. 인코스는 거리상 이점이 있다.
  - 바로 앞이 막히면 **빈 옆 레인(아웃코스)으로 비켜 추월을 시도**한다.
  - 옆도 막혔으면 **감속(블로킹)** — 갇히면 빠져나오기 전까지 못 지나간다.
  - 추월·블로킹 판정은 근접 거리 검사 + PRNG로 처리한다(강체 물리·반발 없음).
- 스킬이 이 모델과 결합한다: 바나나 = 앞/뒤 표적에 명중 시 미끄러져 멈춤(회피 확률 적용), 우다다 = 레인 이탈(옆길로 샘), 낮잠 = 정지 후 폭주. (밀치기·채가기는 부가 시 결합)
- requestAnimationFrame 기반으로 부드럽게 애니메이션한다.
- 결승선 통과 순서대로 순위를 확정한다(동시 도착은 미세 차이로 자동 분리).

---

## 9. 결과 화면
- 1등(또는 우승 팀)을 강조하고 전체 순위를 보여준다.
- 결과 매핑이 있으면 '순위 → 참가자/팀 → 결과'를 함께 표시한다.
- '다시하기'(같은 구성 재경주), '새 게임'(초기화)을 제공한다.

---

## 10. 기술 스택
- **게임 엔진**(시뮬레이션, 스킬·추월·순위·팀 점수 로직)은 **외부 의존성 없는 순수 JS/TS**로 작성한다. 웹/모바일/서버 재사용을 위함.
- **렌더러·애니메이션은 PixiJS**(2D 렌더링)를 사용해 귀여운 캐릭터, 달리기·스킬 연출, 원형 트랙을 표현한다. CDN 또는 번들러 사용 허용.
- **물리엔진은 쓰지 않는다.** 추월·충돌은 엔진의 게임 규칙으로 처리한다(L1).
- 게임 엔진은 렌더러에 의존하지 않는다. 시각 효과·파티클·이징·트랙 모양은 렌더 레이어에서만 처리하며, **경주 결과(도착 순서·스킬·추월 판정)에는 영향을 주지 않는다.**

---

## 11. 향후 확장 로드맵 (지금 구현 X, 설계로 막지 말 것)
- **캐릭터 부가**: 카탈로그의 동물을 데이터(+필요 시 핸들러)로 계속 추가.
- **모바일 앱 전환**: 동일한 게임 로직을 웹/모바일에서 재사용.
- **캐릭터 커스터마이징**: 캐릭터·이름 선택 + 스킨(색상/패턴/액세서리). 캐릭터 데이터가 기본형, 스킨은 그 위에 얹음.
- **멀티플레이어 룸**: 호스트가 방을 열면 다른 사람들이 각자 기기로 참여해 캐릭터를 고르고, 호스트가 출발시키면 모두가 같은 경주를 본다.

---

## 12. 설계 원칙 (확장의 토대 — 1차부터 지킬 것)
1. **로직과 렌더링 분리**
   - 순수 게임 엔진(상태·추월·스킬·순위)을 DOM/UI/PixiJS와 완전히 분리한다.
   - 엔진 입력 = (참가자 설정 + 시드), 출력 = 프레임별 상태와 최종 순위. 엔진은 화면·플랫폼·트랙 모양을 모른다.
   - 렌더러는 엔진 상태를 그리기만 한다.
2. **시드 기반 시뮬레이션 (권장 — 불가침은 아님)**
   - 무작위성은 시드 PRNG로 처리한다. 같은 (설정 + 시드)는 같은 경주를 재생할 수 있게 한다.
   - 멀티플레이어 동기화·시각 검증을 쉽게 하기 위한 것이며, 게임플레이를 제약하면 우선순위를 낮춘다. (L1 추월은 물리엔진 없이 규칙+PRNG라 이 속성을 자연히 유지.)
   - 결정론을 포기할 경우, 멀티플레이어는 시드 공유 대신 **'권위 계산 + 상태/리플레이 브로드캐스트'**로 전환한다.
3. **데이터 주도 + 스킬 핸들러 레지스트리**
   - 캐릭터·게임 모드·스킨은 설정 객체(JSON)로 정의한다.
   - 스킬 동작은 `type → handler` 레지스트리로 등록한다. 새 캐릭터 추가 = 데이터(+새 스킬이면 핸들러 1개).
4. **상태 직렬화**
   - 방 상태(참가자, 캐릭터, 스킨, 모드, 결과 매핑, 시드)를 하나의 JSON으로 export/import할 수 있다.
5. **네트워크·룸 추상화**
   - 참가자 참여/동기화를 트랜스포트 인터페이스(예: `RoomTransport`)로 추상화한다.
   - 1차에는 단일 기기용 `LocalTransport`만 구현하되, `WebSocketTransport`로 교체 가능한 형태로 둔다.
   - 호스트/게스트 역할을 구분한다.

> 권장 작업 순서: **엔진 ↔ 렌더러 ↔ 트랜스포트 인터페이스 + 캐릭터/스킬 스키마를 먼저 정의**하고 들어간다(contract-first).

## 13. 개발 시 시각 검증 (필수)
- 프론트엔드를 수정할 때마다 **Playwright로 실제 브라우저 렌더링 화면을 스크린샷으로 캡처해 눈으로 확인**한 뒤 '완료'로 간주한다.
- 검증 대상 분리:
  - **DOM 셸**(입력칸·버튼·설정 패널): 레이아웃·정렬·잘림·오버플로우.
  - **PixiJS 경주 화면(캔버스)**: 접근성 트리로 캔버스 내용을 볼 수 없으므로 **반드시 스크린샷 이미지로 검증**.
- 주요 상태 캡처: 시작 화면 / 카운트다운 / 경주 중(**각 스킬 발동 순간 — 우다다·낮잠·바나나 명중/회피, 이펙트·대사 말풍선이 보이는지**) / 추월·블로킹 / 곡선 주로 / 결승선 / 결과 화면.
- 시드를 고정해 같은 화면을 재현하면 회귀 검증이 쉽다.
- **원형/자유 주행 체크**: 곡선에서 캐릭터가 작아져 식별이 어렵지 않은지, 겹쳐도 이름표·색 마커가 보이는지, **스킬 대사 말풍선이 동시다발 상황에서도 읽히는지** 확인.
- 뷰포트: 모바일 세로(예: 390×844)와 데스크톱 모두. "3초 안에 출발" 원칙이 모바일 첫 화면에서 지켜지는지 확인.
- 자가 수정 루프: 캡처 → 깨진 부분 발견 → 수정 → 재캡처를 통과할 때까지 반복.

## 14. 품질 기준
- 모바일까지 반응형으로 동작한다.
- 키보드로 조작 가능하고 포커스가 보인다.
- `prefers-reduced-motion`을 존중한다(애니메이션 최소화 옵션).
- 스킬·추월·랜덤 로직이 특정 레인/캐릭터/팀에 편향되지 않아야 한다.

## 15. 1차 개발 스코프 (이 순서로 진행)
1. **계약 정의**: 엔진/렌더러/트랜스포트 인터페이스 + 캐릭터·스킬 스키마.
2. **엔진**: PRNG, 진행도+레인 시뮬레이션, **L1 추월/블로킹 규칙**, 스킬 핸들러 3종(`zoomies`·`nap`·`banana`), 순위 산출.
3. **렌더러(PixiJS)**: 원형 트랙, 기본 3종(`parts` 비주얼, 레퍼런스 SVG 기준), 이름표·색 마커, 달리기·스킬 연출, **대사 말풍선**.
4. **셸 UI**: 이름 입력 → (캐릭터 선택/랜덤) → 출발 → 결과(3초 컷 원칙).
5. **추첨**: 결과 매핑(선택) 연결.
6. **검증**: Playwright 시각 검증 루프 통과(스킬 발동·대사·추월이 화면에서 읽히는지 중점).
7. 이후 카탈로그에서 고슴도치·독수리를 데이터로 **부가**, 팀전·멀티·스킨은 확장 단계.
